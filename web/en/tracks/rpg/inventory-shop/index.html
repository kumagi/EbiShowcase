DEEP DIVE / INVENTORY & SHOP

Every shop is just
three checks

Shop logic has three sequential gates: ① already owned? → reject, ② enough gold? → reject if not, ③ branch by slot type → toggle a bool for equipment, increment a counter for consumables. Nail these three steps and you can build any shop in any RPG.

1

Define items as a slice of structs

Bundling name, slot, price, and stat bonuses into a struct lets you loop through the shop list for both rendering and purchase logic without duplicating data.

shop = []item{{"Coral Sword", "weapon", 60, ...}}
2

Equipment slots as booleans

Weapon and armor each get a single bool. Once sword = true, any later purchase attempt hits the "already owned" gate. Consumables use a counter instead because you can own more than one.

g.sword = true
3

Deduct gold after all checks pass

Only subtract gold after both ownership and affordability checks succeed. Deducting first and checking afterward risks silent bugs where gold goes negative without the purchase completing.

g.gold -= it.price

THE PURCHASE RULE

owned := (choice == 0 && g.sword) || (choice == 1 && g.shield) gate 1: already owned? if owned { message = "already equipped" } gate 2: enough gold? else if g.gold < it.price { message = "not enough gold" } gate 3: purchase succeeds else { g.gold -= it.price; ... }

After passing both gates, a switch choice branches on slot type. Weapon sets a bool, armor sets another bool, consumable increments a counter — each slot stores state in the shape that makes sense for it.

IN THE REAL GAME

Purchase check and slot update

When the player selects an item, the code runs the two gate checks in order. On success it deducts gold and updates the matching slot. The switch keeps each slot type's update logic isolated and easy to extend.

owned := (choice == 0 && g.sword) || (choice == 1 && g.shield)
if owned {
    g.message = it.name + " is already equipped."
} else if g.gold < it.price {
    g.message = "Not enough gold."
} else {
    g.gold -= it.price
    switch choice {
    case 0: g.sword = true
    case 1: g.shield = true
    case 2: g.potions++
    }
    g.message = "Bought " + it.name + "!"
}

WITHOUT SLOT CHECK

No check means duplicate purchases

Without the ownership gate the player can buy the same sword three times, spending gold with no benefit. One boolean prevents the whole problem.

WITH SLOTS

Slot types make extensions clean

Using bools for equipment and counters for consumables from the start makes future features — like selling back items or wearing two rings — straightforward to add without restructuring the data.

YOUR FIRST TUNING

Add a new item to the shop

Append one more struct literal to the shop slice: a "Magic Staff" for 80 gold in the "weapon" slot with attack: 15. The loop-driven rendering and purchase logic will pick it up automatically.
