DEEP DIVE / ACTIVE INDEX

Keep every party member and change only “who fights now.”

All three remain in the party slice and keep their own HP. The front is not a second copy; it is an active index. Switching changes that index to a reserve, but a planned switch and a replacement after fainting spend turns differently.

1

Point to the front by index

Only party[active] attacks and receives counters. Reserve HP stays stored and continues where it left off when that member returns.

front := &party[active]
2

A planned switch spends an action

Selecting a healthy reserve changes active, increments turn, and lets the enemy counter. A better matchup has a time cost.

active = reserve; turn++; enemyAttack()
3

HP zero enters forced-switch state

When the front faints, stop attacks and set mustSwitch. Normal turns cannot continue until a living reserve is selected.

if front.hp == 0 { mustSwitch = true }

IN THE REAL GAME

One switch function handles two transitions

A forced switch clears state without another enemy action. A planned switch spends a turn. One entrance also keeps the “cannot select HP zero” check in one place.

func switchTo(next int) {
    if party[next].hp == 0 { return }
    active = next
    if mustSwitch {
        mustSwitch = false // forced replacement
        return
    }
    turn++                // planned switch
    enemyAttack()
}

WHY KEEP RESERVES?

Remember their damage

Reserves stay in the party array, so they return with the correct remaining HP.

WHY SPEND A TURN?

Make switching strategic

If switching were always free, every turn would simply choose the strongest type. A counter makes timing matter.
