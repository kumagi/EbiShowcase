DEEP DIVE / DELTA TIME

Why do sweets keep rising
even when you do nothing?

Every Update call computes dt — the number of seconds since the last frame — then adds machines × dt to sweets. Because the math is based on seconds rather than frame count, the production rate stays identical whether the game runs at 60 fps or 30 fps.

1

Delta time (dt)

Store the time of the last Update in last. At the top of each Update, subtract last from now to get the elapsed seconds. At 60 fps this is roughly 0.0167 s; at a slower 30 fps it is about 0.033 s.

dt := now.Sub(g.last).Seconds()
2

Cookies per second (CPS)

Define "one machine produces one sweet per second." Then the per-frame contribution is machines × dt sweets. More machines means a proportionally higher rate, which the display shows as a live CPS readout.

g.sweets += float64(g.machines) * dt
3

Delta time cap

When a browser tab is hidden, Update may not be called for a long time. When the tab is shown again, dt could be several seconds, causing a sudden spike. Capping at 0.25 seconds prevents one hidden-tab visit from producing hundreds of sweets.

if dt > .25 { dt = .25 }

THE IDLE FORMULA

gain = machines × dtexample: 3 machines, dt = 0.0167 s0.0167 × 3 ≈ 0.050 sweets/frame

Across 60 frames that is 0.050 × 60 = 3.0 sweets per second. Machine count equals CPS — easy to reason about and display.

IN THE REAL GAME

Record time → compute dt → produce

Grab time.Now() at the top of Update, subtract g.last, cap at 0.25, then multiply by machines. After production, overwrite g.last with now so the next frame gets a fresh dt.

now := time.Now()
dt := now.Sub(g.last).Seconds()
g.last = now
if dt > .25 {
    dt = .25
}
g.sweets += float64(g.machines) * dt

WITHOUT DELTA TIME

Frame rate changes production rate

Adding a fixed amount per frame ties production to fps. A phone running at 30 fps earns half the sweets of a desktop at 60 fps, breaking fairness and feel.

WITH DELTA TIME

Consistent across all devices

Multiplying by elapsed seconds makes production device-agnostic. A slow phone and a fast desktop produce the same number of sweets per real-world second.

YOUR FIRST TUNING

Change CPS multiplier and cost growth

Replace the implicit CPS of 1 per machine with 2 by writing float64(g.machines) * 2 * dt. Adjust the cost increase += 10 to rebalance.
