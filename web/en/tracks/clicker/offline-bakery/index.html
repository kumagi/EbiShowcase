DEEP DIVE / SAVE & OFFLINE

How do sweets keep growing
while the page is closed?

When the game saves, it writes sweets, oven count, and a Unix timestamp to localStorage. On the next load it reads those values back, subtracts the saved timestamp from the current time, and multiplies the elapsed seconds by the production rate to award offline earnings in one go.

1

localStorage via syscall/js

WebAssembly code can call browser APIs through Go's syscall/js package. js.Global().Get("localStorage") returns the storage object, and Call("setItem", key, value) / Call("getItem", key) read and write string values.

s.Call("setItem", prefix+"sweets", value)
2

Offline elapsed time

The saved timestamp is a Unix second count. Subtracting it from time.Now().Unix() gives the seconds the page was closed. That duration is capped at 8 hours (28 800 s) to prevent returning after weeks and instantly winning.

away := math.Min(8*3600, float64(now-stamp))
3

Autosave every 120 frames

Saving on every frame would hammer localStorage unnecessarily. A frame counter modulo 120 triggers a save roughly every two seconds — frequent enough that a browser crash loses little progress, rare enough to be cheap.

if g.frame%120 == 0 { g.save() }

THE OFFLINE RULE

offline = min(away_seconds, 28800) × rate()on loadg.sweets += offline

rate() returns ovens × 5 sweets per second. The offline bonus is a single multiplication applied once at startup, not spread across frames.

IN THE REAL GAME

Load → compute offline → award

load() is called once at game creation. It reads all saved values from localStorage, reconstructs cost from the saved oven count, and awards the offline production lump sum. A non-string getItem result means no save exists yet.

func (g *game) load() {
    store := js.Global().Get("localStorage")
    raw := store.Call("getItem", prefix+"sweets")
    if raw.Type() != js.TypeString { return }
    g.sweets, _ = strconv.ParseFloat(raw.String(), 64)
    g.ovens, _ = strconv.Atoi(
        store.Call("getItem", prefix+"ovens").String())
    g.cost = 20 * math.Pow(1.25, float64(g.ovens))
    stamp, _ := strconv.ParseInt(
        store.Call("getItem", prefix+"time").String(), 10, 64)
    away := math.Min(8*3600,
        float64(time.Now().Unix()-stamp))
    if away > 0 {
        g.offline = away * g.rate()
        g.sweets += g.offline
    }
}

WITHOUT SAVE

Every reload resets progress

No persistence means players lose all ovens on refresh. The core idle-game loop — return after time away to see growth — is impossible without a save system.

WITH OFFLINE CAP

Months away cannot trivialize the game

Without a cap, a player returning after three months would earn millions of sweets in a single load, skipping all mid-game content. Eight hours of offline earnings gives a satisfying but bounded welcome-back bonus.

YOUR FIRST TUNING

Adjust the offline cap

Change 8*3600 to a different hour count to set a new maximum offline bonus. Reduce the autosave interval from 120 frames to 60 for more frequent saves.
