DEEP DIVE / CAMERA SCROLL

How does a small screen
show a huge world?

All game logic uses world coordinates. When drawing, subtract the camera offset to get screen coordinates. The camera smoothly chases a target slightly ahead of the player. Objects that fall outside the screen window are skipped entirely — a technique called culling. These three ideas together make a scrolling platformer.

1

World coordinates

Physics and collision always use world X values (0–2470). The screen width never appears in the update logic — only in draw and camera code.

g.p.x = clamp(g.p.x+g.vx, 0, 2470)
2

Smooth camera follow

Each frame the camera moves 8% of the distance toward its target (a point slightly behind the player). The percentage approach produces natural easing with no extra math.

camera += (target-camera)*.08
3

Culling

Before drawing each object, compute its screen X and skip it with continue if it is fully off screen. Invisible objects cost no draw calls.

if x+w < 0 || x > width { continue }

THE ONE-LINE RULE

screenX = worldX - camera that's it // collision logic stays in world space

One subtraction bridges world and screen. Collision detection never needs to know about the camera — it always works in world coordinates, so moving the camera costs zero changes to the update logic.

IN THE REAL GAME

Camera update and
culled drawing

Update runs all physics in world space and advances the camera. Draw converts each object to screen space and skips anything off screen.

// Update: ease camera toward player
target := g.p.x - width*.4
g.camera = clamp(g.camera+(target-g.camera)*.08, 0, 2020)

// Draw: convert and cull
for _, b := range g.grounds {
    x := b.x - g.camera // world → screen
    if x+b.w < 0 || x > width {
        continue // off screen, skip
    }
    vector.DrawFilledRect(s, float32(x), float32(b.y), ...)
}

WITHOUT WORLD COORDS

You'd rewrite every object when scrolling

If you used screen coordinates exclusively, every object's position would need updating whenever the camera moves. World coordinates make scrolling a one-variable change.

WITH SMOOTH CAMERA

Easing feels natural, not robotic

Moving 8% of the gap each frame means the camera slows as it catches up. Fast movements keep the player on screen; direction changes don't snap the view.

YOUR FIRST TUNING

Change the camera lag factor

Raise .08 toward 1.0 for a camera that snaps instantly to the player. Lower it toward 0.01 for a dreamy lag that reveals upcoming terrain slowly.
