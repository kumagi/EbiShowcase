QUEUE THE NEXT CORNER

Buffered Corner Turns

When the game remembers a direction pressed slightly early, the runner can take it cleanly at the next tile center.

★★☆☆☆PLAYABLE

PLAY FIRST

Cross ten glowing gates

Queue the next direction early with arrows/WASD, click, or touch. Compare the CURRENT and QUEUED readouts.

DEEP DIVE

Separate “moving now” from “want to move next”

Turning immediately on input can push the runner into a wall or off a tile center. Keep current and queued as separate state.

01

Store the press

A key or touch changes only queued.

02

Try it at a center

If the queued neighboring tile is open, copy it into current.

03

Continue if blocked

Keep the request waiting and continue straight while the current direction remains open.

TRY IT / CURRENT vs QUEUED

Queue a facing, commit at center

Direction buttons only write queued. Move between tiles, then Arrive at center to copy queued into current if the next tile is open.

Writing current immediately makes mid-tile wall clips common.

THE BUFFER RULE

input → queuedthenat center: current = queued if passable

Split the press moment from the facing commit.

GO + EBITENGINE

Choose direction only at tile centers

Update may remember input every frame, but it changes direction only when a tile is reached.

if dir, ok := directionInput(); ok {
    queued = dir
}
if reachedTileCenter {
    if passable(tile.add(queued)) {
        current = queued
    } else if !passable(tile.add(current)) {
        return // stop at the center
    }
    target = tile.add(current)
}

Why does this feel better?

The player need not hit one exact frame; the game waits briefly for their intention to become legal.

When does the queue expire?

This game keeps it until success. Other games use a short timer to tune the feel.

Challenge

Expire the queue after ten frames and compare it with the persistent version.
