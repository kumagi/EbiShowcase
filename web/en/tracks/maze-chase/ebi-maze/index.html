DEEP DIVE / INTEGRATED UPDATE ORDER

A finished game passes small states in a deliberate order.

Maze, runner, pearls, and guards are separate data. Update processes input buffer, runner motion, pearl collection, enemy mode, junction choice, then contact. This update order keeps rules and visuals consistent.

1

The runner buffers wanted direction

Save input into wanted even during motion. At a center, try wanted; if blocked, continue using current dir.

wanted = input; chooseAtCenter(wanted, dir)
2

Pearls use a separate layer

Delete pellets[tile] without changing walls. When a catch resets positions, collected pearls stay gone.

delete(pellets, player.tile)
3

Mode selects each guard target

Choose a patrol point, current runner, or last seen tile, then score legal junction directions by distance while excluding reverse.

dir = chooseAtJunction(tile, dir, target)

IN THE REAL GAME

Keep each system's responsibility small

Instead of one giant function, readable Update calls runner, guard, and contact functions in order. Each updates its own state.

func (g *game) Update() error {
    g.player.wanted = readInput()
    g.updatePlayer()       // center movement + collect
    for i := range g.guards {
        g.updateGuard(i)   // mode + junction choice
    }
    g.checkContacts()      // after every position changed
    return nil
}

WHY COLLECT BEFORE CONTACT?

Finalize the arrived tile first

Take a pearl on the arrival frame, then test enemy tile contact, making the result deterministic.

WHY KEEP PELLETS?

Progress survives a mistake

A catch resets only runner and guards, preserving the collection layer and meaningful recovery.
