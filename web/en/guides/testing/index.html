COURSE MAP

Five small refactors

We genuinely refactor earlier lessons into testable code. There is no demo: code and case tables are the visuals.

01

Lift the rule out of the screen

Move the circle hit test out of Ebitengine input code and into a tiny function that knows only numbers. The game owns input and drawing; pure logic owns the answer.

READ →02

Turn boundaries into a test table

PERFECT / GREAT / GOOD / MISS has bug-prone edges. Put inputs and expected answers in a table instead of copying the same test many times.

READ →03

Compare the state one tick later

An active-battle RPG adds speed to a gauge every tick. Test the exact transition that becomes READY at 1000, one step at a time.

READ →04

Never reintroduce a fixed bug

Match-three special pieces are hard to reproduce by hand. Freeze the failing board as small data and keep a test that fails before the fix.

READ →05

Make Tests Read Like Examples

Named tests and the AAA pattern—Arrange, Act, Assert—let a reader predict the function's promise without reading its implementation.
