How a game’s insides are split

Building a game means
assembling a world.

A tiny game can keep enemy positions and dialogue in code. As the world grows, it’s easier if you separate “how it works” (systems) from “what exists” (content). Start with the diagram below—see what one game is made of.

A tile map splits the game world into equal-sized cells. Data can say “column 2, row 3 is a path,” and Draw paints the matching picture later.

Array, slice, and “list”

An array such as [4]int has exactly four slots. A slice such as []int is a sequence that is convenient to grow and shrink. When this course casually says “enemy list,” the Go code usually stores it in a slice.

RULES VS CONTENT

Code is the verb.
Data is the noun.

GO CODE / SYSTEMS

  • Start a conversation
  • Calculate damage
  • Run event steps
  • Draw tiles

Reuse the same systems many times.

ASSET DATA / CONTENT

  • The elder’s dialogue
  • A slime’s HP
  • A treasure event
  • The village tile layout

Add content without changing code.

Organize folders by responsibility

Everything connects through IDs. A map’s onEnter points to an event, which references scenario, image, and audio assets.

MAP DATA

GO + EBITENGINE (advanced)

How to pack files
into the program.

Once the ideas above click, here’s an advanced tool: Go’s go:embed can bundle JSON and images inside the program. The same load path works in the browser (WASM). For now, “that tool exists” is enough—come back after the core LEVELS and this section will read easier.

import (
    "embed"
    "encoding/json"
)

//go:embed assets/**/*.json
var assets embed.FS

func loadMap(id string) (MapData, error) {
    b, err := assets.ReadFile(
        "assets/maps/" + id + ".json",
    )
    if err != nil {
        return MapData{}, err
    }

    var data MapData
    err = json.Unmarshal(b, &data)
    return data, err
}

DO NOT MIX THEM

Separate definitions
from current state.

Definition / immutable

EnemyData {
  ID: "slime"
  MaxHP: 8
  Attack: 3
}

The rulebook. It normally stays unchanged after loading.

Runtime / mutable

EnemyState {
  DataID: "slime"
  HP: 5
  X: 120
  Y: 80
}

A live instance that changes through damage and movement.

Save / only what is needed

SaveData {
  MapID: "village"
  PlayerX: 2
  Flags: ["met_elder"]
}

Save only the differences needed to resume, not every asset.

EVENT INTERPRETER

Play event data
as small commands.

Implement a small set of commands such as say, move, wait, and flag in Go. A scenario is a sequence of those commands.

  1. 01Read the current commandsteps[index]
  2. 02Start its matching actionsay → show a message
  3. 03Wait in Update until completeinput or movement finishes
  4. 04Advance to the next commandindex++

PRACTICAL RULES

Five rules that keep
a growing game healthy.

01

Connect with IDs

Reference slime or event.village.arrive instead of linking files directly.

02

Validate while loading

Reject missing image IDs, invalid tiles, and duplicate IDs before play begins.

03

Keep behavior out of data

Let JSON describe content; keep complex calculations in type-safe Go code.

04

Start with a small format

JSON is enough at first. Add CSV, map editors, or custom tools only when needed.

05

Test assets too

Load every map, event, and translation key automatically to catch broken references.

THE BIG IDEA

A game is not one giant program.
It is a composition of parts, played by systems.

Ebitengine advances the world each frame. System code defines the rules, and asset data supplies the content. This separation lets more people add maps and stories without rewriting the engine.
