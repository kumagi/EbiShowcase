How a game’s insides are split

Building a game means
assembling a world.

A tiny game can keep enemy positions and dialogue in code. As the world grows, it’s easier if you separate “how it works” (systems) from “what exists” (content). Start with the diagram below—see what one game is made of.

GO + EBITENGINE (advanced)

How to pack files
into the program.

Once the ideas above click, here’s an advanced tool: Go’s go:embed can bundle JSON and images inside the program. The same load path works in the browser (WASM). For now, “that tool exists” is enough—come back after the core LEVELS and this section will read easier.

import (
    "embed"
    "encoding/json"
)

//go:embed assets/**/*.json
var assets embed.FS

func loadMap(id string) (MapData, error) {
    b, err := assets.ReadFile(
        "assets/maps/" + id + ".json",
    )
    if err != nil {
        return MapData{}, err
    }

    var data MapData
    err = json.Unmarshal(b, &data)
    return data, err
}

DO NOT MIX THEM

Separate definitions
from current state.

Definition / immutable

EnemyData {
  ID: "slime"
  MaxHP: 8
  Attack: 3
}

The rulebook. It normally stays unchanged after loading.

Runtime / mutable

EnemyState {
  DataID: "slime"
  HP: 5
  X: 120
  Y: 80
}

A live instance that changes through damage and movement.

Save / only what is needed

SaveData {
  MapID: "village"
  PlayerX: 2
  Flags: ["met_elder"]
}

Save only the differences needed to resume, not every asset.

THE GAME IS IN THE PLACEMENT

A game's substance is
an accumulation of deliberately placed assets.

Move a platform half a step farther away. Delay a fighting game's heavy attack by two frames. Bend a tower-defense route to buy the player time to think. The rhythm of difficulty, surprise, tension, and reward comes not from an image alone, but from where and when images, sounds, values, spaces, and events are placed.

Assets are therefore not decoration added at the end. Levels, moves, waves, dialogue, and events are authored data—the material of the work itself. Implementing the rules creates a vessel; it does not yet create the experience inside it.

RULES / CODE

Define what is possible

Run, jump, attack, and advance. Go implements laws that remain dependable.

CONTENT / ASSETS

Shape the experience

Platforms, hitboxes, routes, enemies, and rewards author meaning through placement and values.

EDITOR / AUTHORING

Give creators a workplace

Game-specific words and visuals let non-programmers edit content safely.

RUNTIME / SAVE

Play the authored work

Load validated data, then change and save only the current state.

SPACEwhere→PLACEMENTwhat→PRESSUREwhen→REWARDwhy

Program code makes possibilities. Game data decides the order in which players experience them.

PROGRAMMER'S RESPONSIBILITY

Do not stop at
“we made JSON.”

JSON is readable to a machine, but it may not match how a level designer thinks. A coordinate table hides a broken floor; a frame-count table hides the rhythm of a move. A programmer should design not only the format and runtime, but also a dedicated editor in which creators can think in the language of the game.

01

Direct manipulation

Paint floors, connect paths, and stretch move phases while looking at the subject itself.

02

Constraints and safe defaults

One spawn, positive frame totals, and other rules make invalid work hard to create.

03

Immediate preview

Play changes in the same view and, whenever possible, in the real game.

04

Undo and redo

Fearless experimentation changes both production speed and the number of ideas discovered.

05

Round-trip saving

Load, edit, save, and reload without changing the meaning of the asset.

06

Actionable validation

Say which cell, move, or wave needs repair—not merely that loading failed.

07

Diffs and migration

Keep reviewable text diffs and migrate older schemaVersion values safely.

08

Production playtesting

Do not stop at an approximate preview; test through the product's actual loader.

ONE CONTRACT

The editor and game share one type and one validator.

Different input routes meet at one boundary. Do not create two specifications where an asset saves successfully but breaks only in the game.

THREE GAME-SPECIFIC EDITORS

Turn finished games
into tools for creators.

Do not reduce every genre to a generic JSON form. Put each game's distinct way of thinking on screen. These compact examples really edit, validate, copy, download, and import JSON. Changes persist in this browser.

01 / SIDE-SCROLLING ACTION

Paint terrain. Place events.

Paint floors and ledges into cells; place coins, spawn, goal, and events where they appear. A shared board reveals visual/collision mismatches such as falling through visible ground—something a coordinate table obscures.

  • Generate rendering and collision from the same terrain cells
  • Require exactly one spawn and one goal
  • Check screen edges, jump distance, and collection order at scale
  • Visualize event conditions and trigger areas
Play the finished platformer →

02 / FIGHTING GAME

Tune a move in time and space.

A fighting game's feel comes from startup, active, and recovery frames combined with its hitbox. Show timing and reach together so the designer can build counterplay instead of making button mashing the universal answer.

  • Color startup, active, and recovery on a timeline
  • Preview the current pose and attack hitbox
  • Compare damage, reach, and frame disadvantage together
  • Validate every move for every fighter with one schema
Play the finished fighting game →

03 / TOWER DEFENSE

See routes, build sites, and waves together.

An enemy route is time; a build site is a player choice. Put the board and wave controls in one editor so a creator can tune when, where, and how much pressure arrives.

  • Accept only a continuous route from entrance to base
  • Reject invalid placement such as building on the route
  • Validate enemy count, spawn interval, and starting resources immediately
  • Move between numbers and the board to find impossible or boring stretches

PRACTICAL RULES

Five rules that keep
a growing game healthy.

01

Connect with IDs

Reference slime or event.village.arrive instead of linking files directly.

02

Validate while loading

Reject missing image IDs, invalid tiles, and duplicate IDs before play begins.

03

Keep behavior out of data

Let JSON describe content; keep complex calculations in type-safe Go code.

04

Grow JSON into a dedicated editor

JSON is fine for the first stage. Once people repeatedly create the same kind of level, move, or wave, grow a game-specific editor.

05

Test assets too

Load every map, event, and translation key automatically to catch broken references.

DEFINITION OF DONE

At this point,
the programmer's production line is complete.

“Done” does not mean the programmer personally authored every piece of content. It means designers, artists, and writers can create, test, revise, and ship the work safely without editing program code.

  1. 01The vocabulary is dataLevels, moves, waves, and events have types independent of program code.
  2. 02The game plays it faithfullyEditor and product share the same loader, coordinates, and validation.
  3. 03Creators edit visuallyThey place subjects directly and tune time and space intuitively.
  4. 04Experimentation is safeUndo/redo, defaults, constraints, and actionable errors are present.
  5. 05Files round-tripLoad, edit, and save preserve meaning and produce reviewable diffs.
  6. 06Changes play in the real gameOne action launches the exact scene for testing.
  7. 07Automation prevents breakageTests cover every asset's schema, references, boundaries, and round trip.
  8. 08The next creator can take overDocument controls, constraints, and publishing so production continues without the programmer.

THE BIG IDEA

A game is not merely written.
Its authoring environment is designed.

Ebitengine advances the world each tick. System code defines what is possible, asset data sequences the experience, and dedicated editors turn human judgment into the work itself. Only after this production line exists can a game leave one programmer's hands and continue to grow.

The same principle becomes different game vocabularies: space, time, routes, and story.
