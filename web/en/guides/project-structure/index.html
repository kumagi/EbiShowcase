SYSTEM GUIDE / PROJECT STRUCTURE

When a game grows,
give each job its own room.

`cmd` starts the game, `internal` holds its machinery, and `assets` holds images, sounds, and maps.

examples/project-structure/
├── cmd/reef-run/main.go
├── internal/game/game.go
├── internal/rules/score.go
└── assets/entry / presentation / pure rules / assets

A real split sample

A tiny Space-to-collect-gem game is actually split this way. `rules.Collect` imports no Ebitengine, so `go test ./examples/project-structure/internal/rules` checks it without opening a window.

Keep `cmd` thin. Let `internal/game` connect input, update, and drawing. Move calculations into `internal/rules`. Put assets in `assets`, with their licenses beside them.
