DEEP DIVE / MANY OBJECTS

How can so many bricks
share one rule?

A brick is a tiny record: position plus whether it is still alive. Put those records in a slice and inspect them with the same loop every frame. Repeat one rule many times — that is the core of multi-object games.

1

One record

Each brick is a struct with x, y, and alive. Drawing happens later in Draw.

type brick struct
2

A grid of them

Nested loops fill rows and columns, appending into a slice. Forty-eight bricks do not mean forty-eight variables.

bricks []brick
3

Breaking

Only overlapping bricks set alive = false. Update and draw look only at living ones.

alive = false

THE GRID RECIPE

for row := 0; row < 6; row++innerfor col := 0; col < 8; col++

Place with x = origin + col * width and y = origin + row * height. Same idea as spreadsheet cells.

IN THE REAL GAME

Test collisions inside the loop

After moving the ball, walk every brick. On a hit, destroy it, reverse vy, and stop for that frame.

for i := range g.bricks {
    b := &g.bricks[i]
    if !b.alive {
        continue
    }
    if hitBrick(g.bx, g.by, b) {
        b.alive = false
        g.score += 10
        g.vy = -g.vy
        break
    }
}

WITHOUT A SLICE

brick1, brick2, brick3…

Named variables run out fast. Changing the count means rewriting code.

WITH A SLICE

Count is data; the rule is one

Ten bricks or a hundred bricks still share the same update and draw loops.

YOUR FIRST TUNING

Change rows, columns, and lives

More rows densify the wall. Ball speed and paddle width change the feel immediately.
