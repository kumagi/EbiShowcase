DEEP DIVE / NORMALIZED MOVEMENT

斜め移動が
速くならないのはなぜ？

右と下を同時に押すと、X方向に1、Y方向に1、合わせた長さは √2 ≈ 1.41 になります。そのまま足すと斜めだけ速くなるので、長さで割って「長さ1の矢印」に直してから速さを掛けます。これを正規化と呼びます。

1

入力ベクトル

上下左右のキーを足して、移動の向きを1本の矢印にします。斜めは縦と横の両方が同時に入った状態です。

dx += 1; dy += 1
2

正規化

矢印の長さを math.Hypot で求めて、縦と横の両方をその長さで割ります。これで長さが必ず1になります。

dx /= math.Hypot(dx, dy)
3

速さを掛ける

長さ1の矢印に速さ（ここでは4）を掛けてから位置に足します。向きはそのまま、大きさだけが変わります。

g.px += dx / l * 4

TRY IT / LAB

斜め移動を正規化しよう

斜め入力をそのまま足すと √2 倍速になります。長さで割るとどの方向も同じ速さです。

本物のゲームの公式を、手でゆっくり回す模型です。

生の速さ5.66
正規化後4.00
公式v / |v|

TRY IT · GO

dx, dy := inputX, inputY
if dx!=0 && dy!=0 { dx, dy = dx*0.707, dy*0.707 }
x += dx * speed; y += dy * speed

—

THE NORMALIZE RULE

l := math.Hypot(dx, dy)次にg.px += dx / l * 4

入力がゼロのとき（何も押していないとき）は l もゼロになります。ゼロ除算を避けるため、必ず if dx != 0 || dy != 0 で確認してから計算します。

IN THE REAL GAME

入力を集めて → 正規化 → 移動

キーとタッチの両方で入力を集め、長さがある場合だけ正規化してから位置に足します。タッチの場合は「指の場所 − 自機の位置」が矢印になります。どちらの入力も同じコードで扱えます。

if ids := ebiten.AppendTouchIDs(nil); len(ids) > 0 {
    x, y := ebiten.TouchPosition(ids[0])
    dx = float64(x) - g.px
    dy = float64(y) - g.py
}
if dx != 0 || dy != 0 {
    l := math.Hypot(dx, dy)
    g.px += dx / l * 4
    g.py += dy / l * 4
}

WITHOUT NORMALIZATION

斜めが速いゲームは不公平

正規化なしだと斜め移動は約1.41倍速くなります。プレイヤーが無意識に斜め移動だけ使う「斜め問題」が起きます。

WITH NORMALIZATION

どの向きも同じ速さ

正規化すると8方向すべてが同じ速さになり、操作が公平になります。Survivors系ゲームの移動の基礎です。

YOUR FIRST TUNING

速さ4を変えてみよう

dx / l * 4 の 4 を 6 にするとすばしこく、2 にするとゆっくりになります。敵の速さ 1.15 と合わせて難易度を調整しましょう。
