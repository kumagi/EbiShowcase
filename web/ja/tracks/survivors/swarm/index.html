DEEP DIVE / OBJECT POOL

なぜ100体でも
処理が重くならないの？

スライスはメモリを動的に確保するため、追加・削除のたびに少し時間がかかります。最初から固定サイズの配列を用意して、倒した敵の「席」にすぐ次の敵を上書きすると、メモリ確保のコストがゼロになります。これをオブジェクトプールと呼びます。

1

固定配列

[100]mob と宣言すると、ちょうど100個の場所が最初からメモリに確保されます。スライスと違い、要素の追加でサイズが変わりません。

mobs [100]mob
2

即時リスポーン

「1体倒す」で席0が N1 に書き換わります。配列の長さは増えず、同じ8席を使い回します。

g.spawn(i)
3

オーラ攻撃

自機の周囲62ピクセルが攻撃範囲です。18フレームごとに範囲内の敵をキルカウントし、すぐ同じ席に再スポーンします。

if d < 62 && frame%18 == 0

TRY IT / OBJECT POOL

倒した席にすぐ次の敵を上書き

固定配列の席を消さず、倒したインデックスへ次の敵を上書きします。

スロット表示と右側のマスが同時に変わります。

TRY IT · GO

mobs [100]mob // fixed seats
mobs[i] = spawn() // overwrite, do not delete

THE POOL RULE

mobs [100]mob // 固定サイズ撃破時g.mobs[i] = newMob() // 上書きリスポーン

配列を使うもう一つの利点はキャッシュ効率です。100個の構造体がメモリ上で連続して並ぶので、CPUが一度に読み込める量が増え、ループが速くなります。

IN THE REAL GAME

上書きリスポーンで常に100体

倒した敵（オーラ範囲内かつ18フレームタイミング）は同じインデックスで spawn を呼びます。spawn はランダムな角度と距離で自機の周りに新しい座標を計算して上書きするだけです。

func (g *game) spawn(i int) {
    a := g.rng.Float64() * math.Pi * 2
    r := 280 + g.rng.Float64()*120
    g.mobs[i] = mob{
        g.px + math.Cos(a)*r,
        g.py + math.Sin(a)*r, true,
    }
}

// 更新ループ内：倒したらその席に再スポーン
if d < 62 && g.frame%18 == 0 {
    g.kills++
    g.spawn(i)
    continue
}

WITHOUT A POOL

append/削除でメモリが揺れる

スライスへの頻繁な追加・削除はガベージコレクタの仕事を増やします。100体を常に生成・破棄すると処理が重くなります。

WITH A POOL

配列は最初から決まっている

固定配列は「席」が常に100個。追加・削除のコストがなく、メモリも断片化しません。大量の敵を扱うゲームの定番技法です。

YOUR FIRST TUNING

100体と18フレームを変えよう

配列サイズ [100]mob を [200]mob にするとより激しくなります。オーラ攻撃の間隔 18 を小さくするとクリアしやすくなります。
