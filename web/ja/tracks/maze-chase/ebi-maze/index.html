DEEP DIVE / INTEGRATED UPDATE ORDER

完成ゲームは、学んだ小さな状態を決まった順で渡す。

迷路、走者、真珠、敵は別のデータです。Updateでは入力予約→走者移動→真珠回収→敵モード→交差点選択→接触の順に処理します。この更新順序が、画面とルールの食い違いを防ぎます。

1

走者はwantedを予約する

移動中でも入力をwantedへ保存します。中心へ着いたらwantedが通れるか試し、だめなら今のdirで直進します。

wanted = input; chooseAtCenter(wanted, dir)
2

真珠は独立レイヤー

壁を変えず、pellets[tile]だけを削除します。捕まって位置が戻っても、すでに取った真珠は復活しません。

delete(pellets, player.tile)
3

敵はmodeからtargetを決める

巡回点、現在の走者、最後に見た場所からtargetを選び、交差点では合法方向を距離で比べ、逆走を除きます。

dir = chooseAtJunction(tile, dir, target)

IN THE REAL GAME

各システムの責任を小さく保つ

一つの巨大な関数にせず、順番を読むUpdateから走者、敵、接触の関数を呼びます。各関数は自分の状態だけを更新します。

func (g *game) Update() error {
    g.player.wanted = readInput()
    g.updatePlayer()       // center movement + collect
    for i := range g.guards {
        g.updateGuard(i)   // mode + junction choice
    }
    g.checkContacts()      // after every position changed
    return nil
}

WHY COLLECT BEFORE CONTACT?

到着したマスを先に確定

中心へ到着したフレームで真珠を取り、その後に敵との同じタイル判定を行うため、結果が毎回同じです。

WHY KEEP PELLETS?

失敗しても進みは残る

接触時は走者と敵だけを初期位置へ戻し、収集レイヤーは保存します。再挑戦の意味が保たれます。
