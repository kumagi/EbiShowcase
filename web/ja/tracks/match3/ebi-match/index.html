DEEP DIVE / LEVEL DATA

ゲームのルールと、ステージの数字を分けよう。

「3個なら消える」「4個以上なら特殊ピースを残す」は全ステージで使うルールです。「12手で650点」はこの面だけのデータです。交換中の見た目は0→1の割合で動かし、接触してから盤面のルールを解決します。

1

ステージを表にする

名前、手数、目標点、最初の盤面を一つのstageに入れます。これはゲームへの注文票です。

stage{moves: 12, targetScore: 650}
2

交換を見せてから解決

2個をなめらかに移動させ、接触後に並びを検査します。3個にならない交換は同じ道を戻り、正しい交換だけ手数を1減らします。

swapT += 0.12; resolveOrReturn()
3

長い並びを力へ変える

4個は縦横ロケット、5個は同じ色を消すウェーブ、L字・T字は3×3爆弾になります。爆風が別の特殊ピースへ届くと、続けて発動します。

special = specialForMatch(board, anchor)

IN THE REAL GAME

データを読み、同じルールを動かす

newGameへ別のstageを渡すだけで、手数も目標も盤面も変わります。Updateの交換ルールはそのままです。

type stage struct {
    moves       int
    targetScore int
    board       [rows][cols]int
}

func newGame(level stage) *game {
    return &game{level: level,
        board: level.board,
        moves: level.moves}
}

WHY DATA?

面を増やしやすい

かんたんな面は手数を多く、むずかしい面は目標点を高くするだけ。ルールのバグを増やしません。

WHY INVALID SWAPS RETURN?

1手の意味を守る

3個並びを作らない交換は元へ戻します。「考えて並びを作る」という遊びがはっきりします。

BONUS / DRAW ONLY

ゲームは同じ。絵だけを、別の作り方にする。

これは上の完成ゲームと同じパッケージから作ったWASMです。UpdateとLayoutには手を加えず、そのまま呼びます。元ゲームを等倍のまま操作でき、右には同じ瞬間の輪郭線と、文字だけでゲームを描くASCII画面のモックを並べます。ゲームのルールと見た目は別々に設計できます。

01

UPDATE

入力とゲームの事実を進める。3画面で共通。

02

LAYOUT

元ゲームの大きさをそのまま返す。

03

DRAW

画像・輪郭線・ASCIIなど、描き方を選ぶ。

THE ONLY BOUNDARY

UpdateとLayoutは、元のゲームへそのまま渡す。

等倍snapshotが入力と一致するゲーム画面です。輪郭版はKageで隣の画素との差だけを線にします。ASCIIモックは、同じようなゲーム場面を文字だけでも伝えられることを示します。描画はルールを変更しません。

func (g *gallery) Update() error {
    return g.inner.Update()
}

func (g *gallery) Layout(w, h int) (int, int) {
    return g.inner.Layout(w, h)
}

func (g *gallery) Draw(screen *ebiten.Image) {
    snapshot := ebiten.NewImage(width, height)
    g.inner.Draw(snapshot)
    screen.DrawImage(snapshot, nil) // native-size playable input surface
    drawEdgeInset(screen, snapshot)
    drawASCIIGameMock(screen)
}
