COURSE MAP

4つの小さな改造

過去の教材を、テストしやすいコードへ本当にリファクタリングします。デモはありません。コードとケース表が、このコースのビジュアルです。

01

画面からルールを取り出す

クリック位置と丸の当たり判定を、Ebitengineを知らない小さな関数へ移します。入力と描画はゲーム側、答えを出す計算は純粋ロジック側です。

読む →02

境目をテストの表にする

タイミングメーターの PERFECT / GREAT / GOOD / MISS は、境目でバグが起きやすいルールです。似た確認をコピーせず、入力と期待する答えを表にします。

読む →03

1フレーム後の状態を比べる

アクティブ戦闘RPGでは、毎フレーム『ゲージ + 素早さ』を計算します。1000に届いた瞬間だけREADYになる状態遷移を、1歩ずつテストします。

読む →04

直したバグを二度と戻さない

3マッチの特殊ピースは、十字消しや連鎖のように手動確認が難しい仕組みです。バグが起きた盤面を小さなデータで再現し、修正前に失敗するテストとして残します。

コードを読む前の用語メモ

Updateは次の一コマの数字を決め、Drawはその数字を絵にします。これがEbitengineのゲームループです。

海老・天次郎の見方
Updateが天次郎の数字を変え、Drawが次の一コマを描きます。

func (g *Game) Update() error { return nil }

func (g *Game)の(g *Game)は、この仕事がGameの値を読むための名札（レシーバー）です。
