ゲームの中身を分ける考え方

ゲームを作るとは、
世界を組み立てること。

小さなゲームなら、敵の位置も会話もプログラムに直接書けます。世界が大きくなると、「どう動くか（仕組み）」と「何があるか（中身）」を分けた方が作りやすくなります。まずは下の図で、1本のゲームがどんな部品でできているかを見てみましょう。

プログラムから見れば、すべてはIDでつながります。マップのonEnterがイベントを指し、イベントがシナリオや画像、音を呼び出します。

GO + EBITENGINE（発展）

ファイルをプログラムに
一緒に入れる方法。

上までの話がわかったら、発展として。Go には go:embed という仕組みがあり、JSON や画像をプログラムの中に同梱できます。ブラウザ版（WASM）でも同じ読み込みコードが使えます。いまは「そういう道具がある」くらいで十分です。コアの LEVEL を終えてから戻ってくると読みやすいです。

import (
    "embed"
    "encoding/json"
)

//go:embed assets/**/*.json
var assets embed.FS

func loadMap(id string) (MapData, error) {
    b, err := assets.ReadFile(
        "assets/maps/" + id + ".json",
    )
    if err != nil {
        return MapData{}, err
    }

    var data MapData
    err = json.Unmarshal(b, &data)
    return data, err
}

DO NOT MIX THEM

設計図と、今の状態を分ける。

定義データ / 変えない

EnemyData {
  ID: "slime"
  MaxHP: 8
  Attack: 3
}

ゲームのルールブック。読み込んだ後は基本的に変更しません。

実行状態 / 変わる

EnemyState {
  DataID: "slime"
  HP: 5
  X: 120
  Y: 80
}

いま画面にいる個体。ダメージや移動によって変化します。

セーブ / 必要な分だけ

SaveData {
  MapID: "village"
  PlayerX: 2
  Flags: ["met_elder"]
}

画像やマップ全部ではなく、再開に必要な差分だけを保存します。

EVENT INTERPRETER

イベントデータを
小さな命令として再生する。

say、move、wait、flagなど少数の命令をGoで実装します。シナリオはその命令を並べるだけです。読む前に index < len(steps) を確認し、空のシナリオでも安全に終了できるようにします。

  1. 01現在の命令を読むif index < len(steps) { step := steps[index] }
  2. 02種類に対応する処理を始めるsay → メッセージ表示
  3. 03完了までUpdateで待つ入力や移動の終了
  4. 04次の命令へ進むindex++（最後なら停止）

THE GAME IS IN THE PLACEMENT

ゲームの実体は、
思慮深く置かれたアセットの集積。

横スクロールの足場を半歩遠くする。格闘ゲームの強攻撃を2フレーム遅くする。タワーディフェンスの道を曲げて、考える時間を生む。プレイヤーが体験する難しさ、驚き、緊張、報酬のリズムは、画像そのものではなく、画像・音・数値・空間・時間をどこへどう置いたかから生まれます。

つまりアセットは「最後に飾る素材」ではありません。レベル、技、ウェーブ、会話、イベントを含む、作品を著作するためのデータです。ルールを実装しただけではゲームの器ができただけで、遊びの実体はまだ入っていません。

RULES / CODE

可能な行動を決める

走る、跳ぶ、攻撃する、敵が進む。壊れない法則をGoで実装します。

CONTENT / ASSETS

体験の順序を決める

足場、判定、経路、敵、報酬。遊びの意味を配置と数値で著作します。

EDITOR / AUTHORING

人が作品を作る場所

ゲーム固有の言葉と図で、非プログラマも安全に内容を編集できます。

RUNTIME / SAVE

作品を再生する

検査済みデータを読み、現在の状態だけを変化・保存します。

SPACE空間→PLACEMENT配置→PRESSURE緊張→REWARD報酬

プログラムは「何が可能か」を作る。ゲームデータは「その可能性を、どんな順序で体験させるか」を作る。

PROGRAMMER'S RESPONSIBILITY

「JSONを用意しました」で、
仕事を終えない。

JSONは機械には読みやすくても、レベルを作る人の思考には合わないことがあります。座標の数値では地形の切れ目が見えず、フレーム数の表だけでは技のリズムが見えません。プログラマーは、データ形式とゲーム本体だけでなく、作品を作る人がゲームの言葉で考えられる専用エディタまで設計します。

01

直接操作

床なら描く、経路ならつなぐ、攻撃なら時間軸を伸ばす。対象を見ながら触れます。

02

制約と安全な初期値

開始地点は1つ、合計フレームは正数など、壊れた作品を作りにくくします。

03

即時プレビュー

編集結果を同じ画面、できれば実際のゲームで直ちに再生します。

04

元に戻す／やり直す

失敗を恐れず試せることは、制作速度と発見の量を大きく変えます。

05

保存の往復

読み込み→編集→保存→再読み込みで意味が変わらないことを保証します。

06

検査と具体的なエラー

「失敗」ではなく、どのマス・技・ウェーブを直すべきか示します。

07

差分と移行

テキストで差分をレビューでき、古いschemaVersionも安全に移行できます。

08

本番でのプレイテスト

エディタだけの簡易表示で終えず、製品と同じローダーで試遊します。

ONE CONTRACT

エディタとゲームは、同じ型・同じ検査を共有する。

入力経路が違っても、境界は1つです。保存後にゲーム側だけで壊れる「二重の仕様」を作りません。

THREE GAME-SPECIFIC EDITORS

完成ゲームを、
作り手の道具へ変える。

万能のJSONフォームではなく、ジャンルごとに違う「考え方」を画面へ写します。以下は実際に編集・検査・JSONのコピー／保存／読み込みができる縮小版です。編集内容はこのブラウザに保存されます。

01 / SIDE-SCROLLING ACTION

地形を描き、出来事を置く。

床や足場はマス目へ描き、コイン、開始、ゴール、イベントは見える場所へ置きます。「何もない所で落ちる」のような表示と当たり判定の不一致を、座標表ではなく同じ盤面で発見できます。

  • 地形レイヤーと当たり判定を同じセルから生成
  • 開始地点とゴールは必ず1つずつ
  • 画面端、飛距離、回収順を実寸で確認
  • イベントの条件と発火範囲を可視化
完成した横スクロールを遊ぶ →

02 / FIGHTING GAME

技を、時間と空間で調整する。

格闘ゲームの手触りは、発生・持続・硬直と攻撃判定の組み合わせです。数字の一覧ではなく時間軸と間合いを同時に見せれば、「連打が常に正解」にならない駆け引きを設計できます。

  • startup / active / recoveryを色分け
  • 現在フレームの姿勢と攻撃判定をプレビュー
  • 威力、間合い、硬直差を一緒に比較
  • 全キャラクター・全技を同じschemaで検査
完成した格闘ゲームを遊ぶ →

03 / TOWER DEFENSE

経路、建設地点、ウェーブを一望する。

敵の経路は時間であり、建設地点はプレイヤーの選択肢です。盤面とウェーブを別ファイルに隠さず、同じ編集画面で「いつ、どこに、どれだけ圧力が来るか」を調整します。

  • 入口から拠点まで連続した経路だけを許可
  • 経路上への建設など無効な配置を拒否
  • 敵数・出現間隔・初期資源を即時に検査
  • 数値と盤面を往復し、詰みと退屈を見つける
完成したタワーディフェンスを遊ぶ →

PRACTICAL RULES

大きくなっても
壊れにくい5つの約束。

01

IDでつなぐ

ファイル同士を直接持たせず、slimeやevent.village.arriveのようなIDで参照します。

02

読み込み時に検査する

存在しない画像ID、範囲外のタイル、重複IDはゲーム開始前にエラーにします。

03

データへ処理を書きすぎない

JSONは内容を表し、複雑な計算や分岐の仕組みは型安全なGoコードへ置きます。

04

JSONから専用エディタへ育てる

最初の1ステージはJSONでも構いません。同じ種類の面・技・ウェーブを繰り返し作る段階で、ゲーム固有の編集器へ育てます。

05

アセットもテストする

全マップ、全イベント、全翻訳キーを自動で読み、参照切れがないことを確認します。

DEFINITION OF DONE

ここまでやれば、
プログラマとしての仕事は一通り完成。

「完成」とは全コンテンツを自分で作り終えた状態ではありません。ゲームデザイナー、アーティスト、ライターが、コードを触らずに安全に作品を作り、試し、直し、出荷できる制作ラインを渡した状態です。

  1. 01語彙がデータになっている面・技・ウェーブ・イベントを、コードから独立した型で表せる。
  2. 02ゲームが忠実に再生するエディタと製品が同じローダー・座標系・検査を使う。
  3. 03視覚的に編集できる対象を見ながら配置し、時間や空間を直感的に調整できる。
  4. 04安全に試行錯誤できるUndo/Redo、初期値、制約、具体的なエラーがある。
  5. 05保存が往復する読み込み→編集→保存で意味が変わらず、差分もレビューできる。
  6. 06実際のゲームで試せるワンクリックでその場面を起動し、プレイ感を確認できる。
  7. 07自動で壊れ方を防ぐ全アセットのschema、参照、境界、往復をテストする。
  8. 08次の作り手へ渡せる操作、制約、公開方法を説明し、プログラマ不在でも制作が進む。

THE BIG IDEA

ゲームは「書き切る」ものではない。
作品を作れる環境まで設計するもの。

Ebitengineがtickごとに世界を動かし、仕組みのコードが可能な行動を決め、アセットデータが体験の順序を与え、専用エディタが人の判断を作品へ変えます。この制作ラインを作り切って初めて、ゲームは一人のプログラマの手を離れて育ち続けられます。

同じ原則が、空間・時間・経路・物語という異なるゲームの語彙に変わります。

横スクロールのレベル →格闘ゲームの技 →防衛ゲームの経路と波 →RPGの会話とイベント →
