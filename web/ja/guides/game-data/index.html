GAME ARCHITECTURE / DATA-DRIVEN DESIGN

ゲームを作るとは、
世界を構成すること。

小さなゲームなら、敵の位置も会話もコードへ直接書けます。しかし世界が広がると、「どう動くか」と「何があるか」を分ける必要があります。

RULES VS CONTENT

コードは「動詞」、
データは「名詞」。

GO CODE / 仕組み

  • 話しかける
  • ダメージを計算する
  • イベントを順番に実行する
  • タイルを描く

同じ仕組みを何度も再利用します。

ASSET DATA / 中身

  • 村長のセリフ
  • スライムのHP
  • 宝箱を開くイベント
  • 村のタイル配置

コードを変えずに作品を増やせます。

A SMALL VILLAGE

村ひとつを、
データから組み立てる。

コードには「JSONを読む」「タイルを並べる」「イベントを実行する」という仕組みだけを書きます。村の形や会話の内容はアセット側へ置きます。

assets/
├── maps/
│   └── village.json
├── scenarios/
│   └── intro.json
├── events/
│   └── village.json
├── images/
│   ├── tiles.png
│   └── characters.png
└── audio/
    ├── village.ogg
    └── treasure.wav

フォルダは役割で分ける

プログラムから見れば、すべてはIDでつながります。マップのonEnterがイベントを指し、イベントがシナリオや画像、音を呼び出します。

MAP DATA

{
  "id": "village",
  "width": 5,
  "tiles": [1,1,1,1,1, 1,0,0,2,1],
  "spawn": {"x": 2, "y": 1},
  "onEnter": "event.village.arrive"
}

EVENT DATA

{
  "id": "event.village.arrive",
  "steps": [
    {"type": "music", "value": "village"},
    {"type": "say", "speaker": "elder",
     "text": "scenario.intro.welcome"},
    {"type": "flag", "key": "met_elder"}
  ]
}

GO + EBITENGINE

go:embedで
WASMへ一緒に入れる。

アセットを埋め込むと、デスクトップ版とWeb版で同じ読み込みコードを使えます。起動時に一度JSONを読み、Goの構造体へ変換します。

import (
    "embed"
    "encoding/json"
)

//go:embed assets/**/*.json
var assets embed.FS

func loadMap(id string) (MapData, error) {
    b, err := assets.ReadFile(
        "assets/maps/" + id + ".json",
    )
    if err != nil {
        return MapData{}, err
    }

    var data MapData
    err = json.Unmarshal(b, &data)
    return data, err
}

DO NOT MIX THEM

設計図と、今の状態を分ける。

定義データ / 変えない

EnemyData {
  ID: "slime"
  MaxHP: 8
  Attack: 3
}

ゲームのルールブック。読み込んだ後は基本的に変更しません。

実行状態 / 変わる

EnemyState {
  DataID: "slime"
  HP: 5
  X: 120
  Y: 80
}

いま画面にいる個体。ダメージや移動によって変化します。

セーブ / 必要な分だけ

SaveData {
  MapID: "village"
  PlayerX: 2
  Flags: ["met_elder"]
}

画像やマップ全部ではなく、再開に必要な差分だけを保存します。

EVENT INTERPRETER

イベントデータを
小さな命令として再生する。

say、move、wait、flagなど少数の命令をGoで実装します。シナリオはその命令を並べるだけです。

  1. 01現在の命令を読むsteps[index]
  2. 02種類に対応する処理を始めるsay → メッセージ表示
  3. 03完了までUpdateで待つ入力や移動の終了
  4. 04次の命令へ進むindex++

PRACTICAL RULES

大きくなっても
壊れにくい5つの約束。

01

IDでつなぐ

ファイル同士を直接持たせず、slimeやevent.village.arriveのようなIDで参照します。

02

読み込み時に検査する

存在しない画像ID、範囲外のタイル、重複IDはゲーム開始前にエラーにします。

03

データへ処理を書きすぎない

JSONは内容を表し、複雑な計算や分岐の仕組みは型安全なGoコードへ置きます。

04

小さな形式から始める

最初はJSONで十分です。必要になってからCSV、マップエディタ、専用ツールを追加します。

05

アセットもテストする

全マップ、全イベント、全翻訳キーを自動で読み、参照切れがないことを確認します。

THE BIG IDEA

ゲームは「書き切る」ものではない。
部品を組み合わせて、再生するもの。

Ebitengineが毎フレーム世界を動かし、仕組みのコードがルールを決め、アセットデータが作品の内容を与えます。この分離ができると、プログラマー以外もマップやシナリオを増やせるようになります。
