ゲームの中身を分ける考え方

ゲームを作るとは、
世界を組み立てること。

小さなゲームなら、敵の位置も会話もプログラムに直接書けます。世界が大きくなると、「どう動くか（仕組み）」と「何があるか（中身）」を分けた方が作りやすくなります。まずは下の図で、1本のゲームがどんな部品でできているかを見てみましょう。

GO + EBITENGINE（発展）

ファイルをプログラムに
一緒に入れる方法。

上までの話がわかったら、発展として。Go には go:embed という仕組みがあり、JSON や画像をプログラムの中に同梱できます。ブラウザ版（WASM）でも同じ読み込みコードが使えます。いまは「そういう道具がある」くらいで十分です。コアの LEVEL を終えてから戻ってくると読みやすいです。

import (
    "embed"
    "encoding/json"
)

//go:embed assets/**/*.json
var assets embed.FS

func loadMap(id string) (MapData, error) {
    b, err := assets.ReadFile(
        "assets/maps/" + id + ".json",
    )
    if err != nil {
        return MapData{}, err
    }

    var data MapData
    err = json.Unmarshal(b, &data)
    return data, err
}

DO NOT MIX THEM

設計図と、今の状態を分ける。

定義データ / 変えない

EnemyData {
  ID: "slime"
  MaxHP: 8
  Attack: 3
}

ゲームのルールブック。読み込んだ後は基本的に変更しません。

実行状態 / 変わる

EnemyState {
  DataID: "slime"
  HP: 5
  X: 120
  Y: 80
}

いま画面にいる個体。ダメージや移動によって変化します。

セーブ / 必要な分だけ

SaveData {
  MapID: "village"
  PlayerX: 2
  Flags: ["met_elder"]
}

画像やマップ全部ではなく、再開に必要な差分だけを保存します。

EVENT INTERPRETER

イベントデータを
小さな命令として再生する。

say、move、wait、flagなど少数の命令をGoで実装します。シナリオはその命令を並べるだけです。読む前に index < len(steps) を確認し、空のシナリオでも安全に終了できるようにします。

  1. 01現在の命令を読むif index < len(steps) { step := steps[index] }
  2. 02種類に対応する処理を始めるsay → メッセージ表示
  3. 03完了までUpdateで待つ入力や移動の終了
  4. 04次の命令へ進むindex++（最後なら停止）

PRACTICAL RULES

大きくなっても
壊れにくい5つの約束。

01

IDでつなぐ

ファイル同士を直接持たせず、slimeやevent.village.arriveのようなIDで参照します。

02

読み込み時に検査する

存在しない画像ID、範囲外のタイル、重複IDはゲーム開始前にエラーにします。

03

データへ処理を書きすぎない

JSONは内容を表し、複雑な計算や分岐の仕組みは型安全なGoコードへ置きます。

04

小さな形式から始める

最初はJSONで十分です。必要になってからCSV、マップエディタ、専用ツールを追加します。

05

アセットもテストする

全マップ、全イベント、全翻訳キーを自動で読み、参照切れがないことを確認します。

THE BIG IDEA

ゲームは「書き切る」ものではない。
部品を組み合わせて、再生するもの。

Ebitengineが毎フレーム世界を動かし、仕組みのコードがルールを決め、アセットデータが作品の内容を与えます。この分離ができると、プログラマー以外もマップやシナリオを増やせるようになります。

この考え方はRPGだけでなく、会話・戦略・カード・探索にも使えます。

RPG・会話・イベント →会話ゲームのシーンデータ →戦略SLGの任務データ →カードゲームのデッキデータ →
