# 手書き教材へ「二層コード + 1ルール」を入れる手順

対象は `web/{ja,en}/tracks/platformer/` のように、生成器ではなく個別 HTML を保守する教材です。目的は、説明用の短い断片を「実際にどのファイルを開くか」と混同させず、読者が一つの変更を試して確認できるようにすることです。

1. まず対応する `games/tracks/<track>/<slug>/main.go` を開く。ページにはこの**実在する入口パス**と、そのままの短いコードを `EDIT THIS ENTRY / 編集する入口` として載せる。
2. 入口が `Run` や設定を渡すだけなら、その呼び先の実在する `internal/...` ファイルから、説明したい `Update` の小さな範囲を選ぶ。これは `HOW IT WORKS / 仕組みの抜粋` と明記し、入口そのものだとは呼ばない。
3. 概念カードは必ず三つに分ける。`DATA` は何を記録するか、`UPDATE` は何をどの順番で変えるか、`DRAW` は状態をどう絵にするかを一つずつ説明する。Draw が状態を変えないことも書く。
4. 既存の一般的な「数値を変えよう」ではなく、`YOUR FIRST RULE` を一つ置く。対象ファイル、変更場所、変更内容、確認方法を一文にする。例: `main.go の speed を変更し、go test ./... の後にブラウザでジャンプ距離を比べる。`
5. JA と EN の両方へ同じ構造を置く。翻訳してもパス・コード・検証コマンドは同じものを保つ。

## Platformer の確認ポイント

`games/tracks/platformer/scrolling-stage/main.go` のような入口と、必要なら内部ロジックを追えるか確認します。ページに表示する断片は、存在しない API や実装を「実際のコード」として見せません。編集後は少なくとも次を実行します。

```sh
go test ./games/tracks/platformer/...
node scripts/check-lessons.mjs
git diff --check
```

ゲームを実行して、変更したルールが `Update` の結果として変わり、`Draw` はその結果を表示しているだけかも見ます。デスクトップと幅の狭い画面で、入口・内部・ルールの三つが順番に読めることを確認してください。
